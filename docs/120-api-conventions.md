## API Conventions & Cross-Cutting Rules

### Purpose

This section defines cross-cutting API conventions that apply to all endpoints.

It establishes:
- Standard request and response patterns
- Pagination, filtering, and sorting rules
- Idempotency and retry behavior
- Correlation and tracing support
- Consistency expectations across all API endpoints

This ensures that all endpoints behave predictably as a unified system rather than a collection of isolated designs.

---

### Core Principle

All API endpoints must follow consistent conventions.

Consistency improves:
- developer experience
- client implementation simplicity
- debugging and observability
- AI-assisted development reliability

The API should feel uniform regardless of which endpoint is being used.

---

## Response Envelope Convention

### Standard Structure

All successful responses should follow a consistent structure.

Conceptual shape:

- data (primary payload)
- meta (optional metadata)
- error (only present for failures)

---

### Success Response

Responses should include:

- `data`: the main response payload
- `meta` (optional): pagination info, counts, or additional context

Responses should avoid mixing metadata into the primary data object.

---

### Error Response

Errors should follow a consistent structure including:

- error type or code
- message (user-safe or developer-safe depending on context)
- optional details for debugging
- optional correlation identifier

Error structure must be consistent across all endpoints.

---

## Pagination Rules

### When to Use Pagination

Pagination should be used for any endpoint returning:

- lists of resources
- historical records
- logs or entries
- potentially large datasets

---

### Pagination Model

The API should support:

- `limit` (number of items to return)
- `cursor` or `offset` (implementation-dependent)

Cursor-based pagination is preferred for:
- stability
- performance at scale

---

### Pagination Response Metadata

Paginated responses should include:

- current cursor or position
- next cursor (if more data exists)
- total count (optional if expensive)

---

## Filtering & Sorting Rules

### Filtering

Endpoints may support filtering via query parameters.

Examples:

- by date range
- by status (Complete, Partial, Missed)
- by plan
- by commitment type

Filtering must be:

- explicitly documented per endpoint
- predictable in behavior
- validated server-side

---

### Sorting

Sorting should be supported where meaningful.

Examples:

- chronological (default for most endpoints)
- reverse chronological
- by priority or defined order

Sorting must be:

- explicitly defined
- stable
- consistent across calls

---

## Idempotency & Retry Behavior

### Core Requirement

Endpoints must behave predictably under retries.

This is critical for:

- mobile networks
- intermittent connectivity
- optimistic UI patterns

---

### Idempotent Operations

Where possible, operations should be idempotent.

Examples:

- setting a value
- marking a day complete
- updating preferences

Repeated identical requests should not produce unintended side effects.

---

### Non-Idempotent Operations

Where operations are inherently non-idempotent (e.g., creating entries):

- the API should tolerate safe retry patterns where possible
- duplicate creation should be prevented where feasible
- the system should favor consistency over duplication

---

## Correlation & Request Tracing

### Purpose

Every request should be traceable across the system.

---

### Correlation Identifier

The system should support a correlation ID for each request.

This ID should:

- be generated by the server if not provided
- be included in logs
- be returned in responses where useful

---

### Usage

Correlation IDs allow:

- tracing a request across services
- debugging production issues
- linking frontend errors to backend logs

---

## Optimistic UI Support

### Principle

The API should support optimistic UI patterns without compromising correctness.

---

### Requirements

If the client performs optimistic updates:

- the server must return authoritative state
- discrepancies must be resolvable
- failures must be clearly surfaced

The API must not silently accept inconsistent state.

---

## Consistency Across Endpoints

All endpoints must:

- follow the same naming conventions
- follow the same response structures
- use consistent error formats
- apply authentication and ownership rules uniformly
- respect editability and business rules consistently

Inconsistency between endpoints is considered a defect.

---

## Validation Rules

All endpoints must:

- validate input thoroughly
- reject invalid or inconsistent requests
- provide clear error messages
- avoid silent correction of invalid input

The system must not “guess” user intent in ambiguous situations.

---

## Performance Considerations

Endpoints should be designed to:

- minimize unnecessary round trips
- return sufficient data for the client to act
- avoid requiring multiple follow-up calls for basic operations

Performance optimizations must not compromise:

- correctness
- clarity
- consistency

---

## Strategic Principle

The API should behave as a cohesive system, not a collection of independent endpoints.

The guiding rule is:

Consistency is a feature.
